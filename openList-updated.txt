1. Problem Statement

Teams constantly juggle at least three separate tools: a document editor for notes, a task manager for assignments, and a messaging app for real-time coordination. The friction of switching between these tools causes context loss, stale information, and coordination failures - especially in remote or hybrid settings.


Core Problem
No single lightweight tool lets a small team create a shared note, convert any line of it into an assigned task with sub- items, upload supporting images, and see every change from teammates in real time - all while working smoothly offline.


2. Target Audience

Remote-first teams of 2-15 people: startups, freelance squads, and small dev teams who share a workspace and need real-time task coordination without paying for enterprise tools. Secondary users are solo power users who want a unified notes-plus-tasks editor that syncs across their devices.



3. Proposed Solution - OpenList

A local-first Flutter app that merges collaborative notetaking with real-time task management inside shared Spaces. Every piece of content - text, heading, checklist item, image - is an Atomic Block that can be converted, reordered, or assigned without losing its history.



Core Concepts
Ģ Shared Spaces - a collaborative workspace joined by multiple users, each with a role (Admin or Member). Notes and task lists live inside Spaces.
Ģ Atomic Blocks - every row inside a note or task is a typed reactive object: text, heading, checklist, image, or sub-task.
Ģ Pinned Items - tasks and notes can be pinned to the dashboard per-user or per-Space for quick access.
Ģ Real-time Push - any block added or modified by one user is broadcast via Supabase Realtime to all active members within sub-100ms.
Ģ Task Completion Flow - completing a task updates the shared progress ring and removes it from everyone's pinned view. An in-app notification is sent to all Space members.
Ģ Admin Reminders - when an Admin removes or reassigns a task, all affected members receive an in-app and push notification and update happens on their end too.

8. System Architecture

Clean layered architecture in Flutter. The UI never touches the network. All layers communicate downward only.



Layer Overview

PresentationFlutter widgets + Riverpod providers + go_router. Read-only from Isar streams. All writes go through use-cases.DomainPure Dart use-case classes. No Flutter, no Supabase, no Isar imports. Business rules live here.DataTwo repository implementations per entity: IsarRepository (local) and SupabaseRepository (remote). Injected via Riverpod.SyncManagerA Dart Isolate running a delta queue. Reads 'pending' records from Isar, upserts them to Supabase, marks them 'synced'. Never blocks the main thread.Realtime LayerA Supabase Realtime subscription per joined Space. On receiving a broadcast, it writes the delta directly to Isar and notifies Riverpod providers to rebuild.

Conflict Resolution
MVP strategy: Last Write Wins via updated_at timestamp. If a remote delta arrives with a newer updated_at than the local Isar record, the remote wins and Isar is overwritten. This is a documented tradeoff - field-level CRDT merging is deferred to V2.

Image Upload Flow
Ģ User picks image via image_picker plugin
Ģ Compressed locally to = 1 MB using flutter_image_compress
Ģ Uploaded to Supabase Storage bucket: spaces/{space_id}/images/{uuid}.jpg
Ģ Public URL (or signed URL for private spaces) written as the block.content of an image block
Ģ Realtime broadcast notifies other Space members; they load the URL from their Isar replica


Notification Architecture
Ģ In-app: A Supabase Realtime channel on the notifications table (filtered by user_id) delivers new rows to the client instantly. Bell badge increments; panel lists unread items.
Ģ Remote push: A Supabase Database Webhook (or Edge Function) fires on notification INSERT. It calls FCM v1 API with the recipient's fcm_token to deliver a push notification when the app is backgrounded.
Ģ Local reminders: flutter_local_notifications schedules a local alarm at reminder_at time stored in Isar. No server dependency - works fully offline.


Authentication & Security
Ģ Supabase Auth issues JWTs on login, refreshed automatically by supabase_flutter client
Ģ Row Level Security (RLS) on all Supabase tables - users can only read/write records belonging to Spaces they are members of
Ģ Supabase Storage bucket policies enforce that only space_members can read/write images in their Space's bucket
Ģ flutter_secure_storage holds the refresh token; never in shared preferences or plain files

9. Offline Working & Sync Strategy


OpenList treats the local device as the primary source of truth at all times. Every write - whether online or offline - goes to Isar first. The UI never waits for the network. A background SyncManager isolate queues pending changes and flushes them to Supabase the moment connectivity is restored, automatically broadcasting updates to all Space members in real time.



Reconnect Flow (Pull-First)
? SyncManager pulls all Space records modified after the device's last synced_at timestamp and writes any
remote-newer records to Isar first.
? Local pending deltas are then upserted to Supabase in created_at order with original updated_at timestamps preserved.
? Supabase Realtime broadcasts confirmed changes to all active Space members - no manual refresh required on any device.


What Works Fully Offline

ActionOffline BehaviourCreate / edit any task, note, or blockSaved to Isar instantly; synced on reconnectCheck a task completeMarked locally; progress ring updates; members notified on reconnectUpload an imageCached locally; upload queued; block shows local path until syncedView all tasks and notesReads from Isar - fully available, zero network dependencyLocal remindersFire via flutter_local_notifications - no server neededReal-time teammate updates / Space invitesRequires connectivity. Realtime subscription resumes and catches up automatically on reconnect.


10. AI-Powered Note-to-Plan Conversion

Users paste raw meeting notes into any Note and tap "Convert to Plan". OpenList sends the text to Google Gemini 2.5 Flash, which extracts tasks, infers due dates, suggests assignees, assigns priorities, and groups items by theme - all returned as structured JSON. The result appears on a Plan Preview screen where the user reviews and edits everything before confirming. Only on confirmation are tasks written to Isar as real Atomic Blocks and broadcast to Space members via Supabase Realtime in the usual <100ms window. The original note is never modified.

Prompt forces JSON-only output, parsed via dart:convert into List<ExtractedTask>.ArchitectureGeminiRepository (Data layer) ? ConvertNoteToTasksUseCase (Domain

Privacy & Consent


Performance & Real-Time Sync
The AI call (~2-4 seconds) and the Supabase Realtime pipeline are completely separate systems on separate threads. The <100ms collaborative sync guarantee is unaffected. The API latency is a one-time, user-initiated wait shown with a loading indicator - it does not touch any collaborative flow.

Suggested assignees from the AI are free-text strings only - never auto-matched to Space members. The user always makes the final assignment from the real member list in the review screen. If JSON parsing fails, the user sees a toast: "Couldn't extract tasks - try simplifying your notes."


Known Limitations

Consent modal and button tooltip warn the user explicitly.Free Tier PrivacyGoogle's free tier may use inputs for training. MVP internal use only. Paid tier required before any external user can access this feature.Assignee MatchingAI suggests names from note text but cannot query Space membership. User manually maps suggestions to real members in the review screen. V2 may add fuzzy matching.
